== Publish on code.dlang.org

The registry does not receive uploads from your PC. You **register** the Git repo once; **versions** come from **git tags** (`v1.2.3` style).

. Create an account: link:https://code.dlang.org/register[code.dlang.org/register].
. Log in and **register** the package with your Git host user/org and repo name (`AMDphreak` + `libequivalence` for this project).
. Ensure `dub.json` has sensible `description`, `license`, `authors`, and `dependencies` (already required for a clean listing).
. Cut releases with **annotated or lightweight tags** named like `v1.0.1` (semver). The registry polls Git periodically (about every ~30 minutes) and picks up new tags automatically.
. After the first successful crawl, the package page will be at `https://code.dlang.org/packages/libequivalence` and dependents can use `dependency "libequivalence" version="~>1.0.0"` (registry) instead of a raw `repository` + commit pin—though you may still pin commits for maximum reproducibility.

Official overview: link:https://dub.pm/dub-guide/publishing/[Publishing packages — DUB documentation].

== Use in DUB

DUB 1.41 and newer require a **commit hash** (not only a semver tag) in the `version` field when using `repository`. Pin the revision you want (for example the commit that `v1.0.0` points to):

[dub.sdl]
----
dependency "libequivalence" version="ecbc77baa6863517b15b2fcd467aa24b2429386a" repository="git+https://github.com/AMDphreak/libequivalence.git"
----

Git tags such as `v1.0.0` / `1.0.0` are still useful for human-readable releases; bump the hash in dependents when you tag a new library version.
